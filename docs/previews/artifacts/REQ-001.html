6. 本文

6.1 文書概要

本文書は、京成電鉄向け WeChat ミニプログラムについて、業務要件、機能要件、外部システム連携、非機能要件および確認事項を整理し、関係者レビューと後続設計の入力とする。

対象は、訪日中国人利用者が WeChat 上で Skyliner 関連乗車券を購入・決済し、顔認証または将来の乗車用 QR により乗車できるサービスである。中間サーバ、管理機能および既存システム連携を含む。

本文書は review_required である。要件、責任分界、API 仕様、リリース日、UAT 条件には未確定事項があり、approved 文書ではない。

6.2 プロジェクト背景

背景・課題

  • 訪日中国人旅行者に対し、追加アプリのインストールを必要としない購入・乗車体験が求められている。
  • 既存の乗車券管理、顔認証、改札、座席管理等を活用しつつ、WeChat ミニプログラムと中間サーバを新設する想定である。
  • 外部 I/F の仕様・提供日、データ取扱い、ベンダー責任分界が未確定であり、設計・見積・日程に影響している。

目的

  • WeChat 内で乗車券の検索、購入、決済、利用状態確認を完結できる導線を整備する。
  • 顔認証および後続の乗車用 QR に対応可能な業務・システム基盤を段階的に構築する。
  • 京成電鉄の公式デジタル接点として、運用に必要な注文・状態・異常情報を管理できるようにする。

対象ユーザー

  • 簡体字中国語を主に利用する訪日中国人旅行者
  • 京成電鉄の駅係員、業務管理者、問い合わせ対応担当者
  • NSS および外部システムベンダーの運用・保守担当者

6.3 対象スコープ

対象範囲

  • WeChat ミニプログラムによる Skyliner 関連乗車券の選択、購入、WeChat Pay 決済
  • 購入後の顔登録 URL 連携、顔認証結果・注文状態の表示候補
  • 簡体字中国語 UI、主要画面・画面遷移、利用者向けエラー・案内表示
  • 中間サーバ、管理機能、開発・検証・本番環境
  • 乗車券管理、顔認証、検札・印刷等の外部 I/F 整理と結合試験支援

対象外範囲

  • QR 改札機による正式な入出場、乗車用 QR の本番発行
  • 座席指定、座席・列車変更等の座席関連例外処理
  • 既存システムの大規模改修
  • 高度な顧客分析、広告、マーケティング自動化
  • 正式な運用保守契約および簡体字中国語以外の多言語展開

Pending scope

  • 顔認証未完了・失敗・システム故障時の有人対応と通常フローとの境界
  • 取消・返金の自動処理、状態反映、手数料および駅係員操作
  • 管理画面の利用者、必要機能、表示データ、マーケティング利用範囲
  • 乗車用 QR、QR 改札、座席指定の対象フェーズ
  • 非機能、セキュリティ、監視、運用保守、SLA/SLO

顧客・ベンダー確認事項

  • 購入可能票種、販売条件、利用日、人数、価格・手数料
  • 顔認証を通常経路とする条件および利用者同意・案内文
  • 取消・返金・交換・有人対応の業務ルールと実装主体
  • 駅係員が参照・操作する情報と監査記録
  • 乗車券管理側が提供可能な状態照会・更新 I/F

6.5 機能要件

WeChatミニプログラム機能

  • 乗車券情報・購入入口、商品選択、注文確認、決済開始
  • WeChat Pay 結果および注文状態の表示
  • 顔登録 URL への接続、認証状態・エラー・再試行案内
  • 利用可能なチケット・注文詳細・利用案内の表示
  • WeChat 審査に必要な利用規約、プライバシー説明、同意導線

中間サーバ機能

  • ミニプログラム向け API と外部 I/F の中継
  • 注文、決済、票券、認証状態の整合管理候補
  • API 認証、エラー処理、タイムアウト、再試行、関連 ID 付きログ
  • 環境別設定および外部接続先の管理

管理画面機能

  • 注文・決済・票券・顔認証・異常状態の検索と参照
  • 問い合わせ・人工対応に必要な履歴の確認
  • 運用監視に必要な最小ステータスおよびエラー情報の表示
  • 利用者、権限、操作履歴、データ出力の要否は確認事項とする。

外部システム連携・審査対応

  • 乗車券管理、顔認証、WeChat Pay、検札・印刷等との連携
  • WeChat ミニプログラムおよび WeChat Pay の審査資料、カテゴリ、外部遷移条件への対応
  • 顔写真・外部 URL・決済機能に対する審査制約を設計前に確認する。

  • API の最終責任、障害一次切り分け、データ補正、返金・人工処理は API-001 と ARC-001 で確定する。

6.9 リスク・課題

主要リスク

  • API 仕様・データ定義の不足または提供遅延により、要件・設計・見積・試験日程が変動する。
  • 顔情報、個人情報、越境データ、WeChat 規約により連携方式が制限される。
  • 中国-日本間通信、外部 API 応答、ピーク利用により利用体験や駅運用へ影響する。
  • 性能、セキュリティ、運用要件が未定義のまま後工程へ進む可能性がある。
  • Ph.1 / Ph.1.1 / Ph.2 の範囲、概算見積、リリース目標が未確定である。

対応方針

  • 現行 I/F 仕様を先行受領し、QA と影響分析を段階的に実施する。
  • scope、API、architecture、WBS を連動して更新し、変更履歴を残す。
  • 個人情報・顔情報は最小化、同意、保存・削除、権限、越境条件をレビューする。
  • 未確定項目は open / candidate / review_required のまま管理し、確認記録なしに解消済みとしない。

  • 票種、価格、販売条件、取消・返金、手数料、状態遷移
  • 管理画面、データ利用、個人情報・顔情報、運用保守の範囲
  • UAT 担当、受入条件、残存障害基準、リリース判定

  • 日立製作所 I/F 仕様書の現行版・正式版とシステム概要図
  • 購入情報以外の状態照会、取消・返金、人工対応状態反映の可否
  • 顔登録 URL、認証結果、エラーコード、障害判定、紙媒体連携
