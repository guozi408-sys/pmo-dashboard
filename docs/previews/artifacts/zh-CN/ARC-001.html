ARC-001 系统架构定義

本書は登録済み source summary、MGT-001、REQ-001 および ACAP METHOD-006 に基づく系统架构レビュー用初版である。外部 I/F、責任分界、環境およびデータ取扱いは未確定事項を含み、顧客レビュー完了までは review_required とする。

1. 文档概要

本文書の目的

京成電鉄向け WeChat ミニプログラムについて、対象システム、システム境界、主要コンポーネント、データ連携、環境および非機能・セキュリティ観点を整理し、要件定義、外部ベンダー協議、API 設計およびテスト計画の入力とする。

対象プロジェクト・前提資料

  • KEISEI WeChat Mini Program Project
  • MGT-001 プロジェクト計画書 v0.1
  • REQ-001 要件定義書 v0.1
  • SRC-001～SRC-010 のうち、本書 frontmatter に記載した KEISEI 事実資料
  • ACAP METHOD-006 keisei-system-boundary-interface-draft.md
  • risk / issue / action / decision log
  • SRC-011 は他案件の様式参考であり、KEISEI のシステム事実には使用しない。

本文書のステータス

review_required。本書は論理構成と確認境界を示すもので、正式な物理設計、API 仕様、ネットワーク設計、実装承認または受入基準ではない。

構成概要とシステム境界

  • WeChatミニプログラム: 利用者向けの情報表示、購入、決済開始、顔認証接続、注文・利用状態表示を担う候補。
  • 中間サーバ: ミニプログラム、管理機能、決済および既存システムを接続する API Hub とデータ整合管理を担う候補。
  • 管理機能: 注文、決済、認証、異常、人工対応に必要な情報を参照する候補。機能・利用者・権限は未確定。
  • 既存システム: 乗車券管理、顔認証、検札・改札、印刷、座席管理を京成電鉄側境界として扱う候補。
  • 外部サービス: WeChat プラットフォーム、WeChat Pay および関連審査・決済基盤。

NSS と京成電鉄・外部ベンダー間の実装、運用、障害対応およびデータ管理責任は API-001 と本書のレビューにより確定する。

参照: SRC-001、SRC-002、SRC-003、SRC-007、METHOD-006

6. 外部システム連携

乗車券管理システム連携

購入情報、注文・票券状態、支払確認、取消・返金、人工対応状態等が候補となる。ただし 2026 年 7 月時点では、日立製作所様の提供範囲が購入情報連携 I/F を中心とする可能性があり、正式 I/F 仕様書を確認する。

座席管理システム連携

座席照会、指定、変更・取消、票券との関連付けは後続フェーズ候補とする。Ph.1 の開始条件には含めない。

顔認証システム連携

顔登録 URL の生成・取得、写真撮影または外部遷移、認証結果、エラーコード、再試行および障害判定が候補となる。URL 生成主体、結果通知先、保存・削除責任は未確定である。

WeChat Pay / 決済連携

支払開始、結果通知、返金、照合を対象候補とする。決済主体、加盟店、通貨、審査、コールバック検証、冪等性および支払成功後の出票失敗対応を確認する。

API仕様確認事項・責任分界

  • API 方向、認証、データ項目、状態コード、タイムアウト、再試行、頻度制限
  • 開発・検証・本番の接続情報、試験データ、ネットワーク、証明書
  • 障害の一次切り分け、データ補正、再処理、監視、問い合わせ窓口
  • 京成電鉄、日立製作所様、その他既存ベンダー、NSS の実装・試験・運用責任

本章は API-001 の入力であり、正式 API 仕様を代替しない。

参照: SRC-001、SRC-002、SRC-003、SRC-007、METHOD-006

9. リスク・課題

  • 外部システム仕様・データ定義・接続情報の不足または提供遅延
  • API 認証、ネットワーク、環境、エラー処理、状態整合の未確定
  • 顔情報・個人情報の保存、削除、越境移転および WeChat 規約上の制約
  • 中国-日本間通信や外部 API 応答による性能・可用性への影響
  • 関係ベンダーの実装、結合試験、UAT、障害対応の責任分界未確定

対応方針は、現行 I/F 仕様を先行受領し、API-001 に QA と責任分界を記録すること、ARC-001・REQ-001・WBS-001 を連動更新すること、未確定事項を candidate/open のまま管理することである。

参照: RISK-001、RISK-002、RISK-003、RISK-004、RISK-005、RISK-007、RISK-013、RISK-014、RISK-015

10. 確認事項

顧客確認事項

  • AWS、ネットワーク、ドメイン、証明書、環境および運用責任
  • 管理機能の利用者、権限、必要データ、監査・保存条件
  • 個人情報・顔情報の同意、保存、削除、越境移転、委託先管理
  • Ph.1 / Ph.1.1 / Ph.2 の正式構成範囲とリリース順序

外部ベンダー確認事項

  • 日立製作所様の I/F 仕様書、システム概要図、接続環境、提供日
  • 購入情報以外の状態照会、取消・返金、認証結果、人工対応状態の連携可否
  • 顔登録 URL、結果通知、エラーコード、再試行、写真・顔情報の取扱い
  • 改札、印刷、座席管理の担当ベンダー、技術窓口、試験条件

NSS内部確認事項

  • 中間サーバの機能分割、状態管理、正本、補正・リカバリ方式
  • 環境設計、非機能目標、監視、ログ、セキュリティ、運用設計
  • 外部 API 不足時の代替運用と見積・スケジュール影響
  • METHOD-006 の候補 I/F を API-001 へ採用する条件
