6. 正文

6.1 设计方针

本设计书旨在整理 KEISEI WeChat 小程序中面向用户界面及管理界面的候选方案。画面 ID、画面名称、使用阶段及跳转路径为基于当前资料的候选方案，将在与客户及相关供应商确认后确定。

画面定义的前提条件

观察视角当前整理内容状态
目标用户访日中国游客用户、京成电铁侧或运营负责人review_required
目标渠道WeChat小程序、管理页面review_required
显示语言用户端主要以简体中文展开探讨candidate
Ph.1以现有 eTicket 相当的购买及显示为中心进行整理candidate
Ph.1.1整理人脸认证用照片拍摄及上传等candidate
Ph.2整理 QR 乘车券、座席指定、乘车 QR 发券为候选candidate
管理页面设想用于用户、交易、车票、错误等的确认用途pending_confirmation

*追溯注记: SRC-011 为其他项目的样式参考，不作为表示 KEISEI 画面需求的事实资料使用。*

6.3 画面跳转

画面跳转流程

  1. 购买流程 (FLOW-001): MP-001 → MP-002 → MP-003 → MP-004 → MP-005 → MP-006
  2. 历史流程 (FLOW-002): MP-001 → MP-007 → MP-006
  3. 人脸认证流程 (FLOW-003): MP-008 → MP-012 → MP-009
  4. 座位指定流程 (FLOW-004): MP-006 → MP-010 → MP-003 → MP-011
  5. 管理员流程 (FLOW-005): ADM-001 → ADM-002 → ADM-003

*确认事项: 分支、返回操作、超时、重复操作、支付中途退出、API 异常、QR 重新展示等为确认中事项。*

---

6.4 画面通用规范

  • 用户端主要探讨简体中文，确认日语及英语的必要性与翻译责任。
  • 考虑到通信与 API 延迟，设计加载、等待与重新确认的 UI。
  • 定义防止支付/发券重复操作、超时、重试及状态不一致时的引导。
  • 定义个人信息与人脸信息的最小化展示、脱敏、权限控制、审计日志与删除策略。
