MGT-001 プロジェクト計画書

本書は ACAP 方法論および登録済み source summary に基づく初版である。未確定事項は候補または確認事項として記載し、顧客レビュー完了までは review_required とする。

1. プロジェクト概要

プロジェクト目的

訪日中国人旅行者に対し、使い慣れた WeChat 上で Skyliner の情報確認、乗車券購入、決済および乗車認証を行えるサービスを構築する。利用者の購入・乗車体験を改善するとともに、京成電鉄の公式デジタル接点と管理基盤を整備する。

背景

  • 訪日中国人旅行者の回復を背景に、独立アプリのインストールを必要としない購入環境が求められている。
  • 既存の乗車券管理、顔認証、改札、座席管理等のシステムを活用しながら、新 WeChat ミニプログラムと中間サーバを構築する。
  • API 仕様、ベンダー責任分界、個人情報・越境データ、性能・運用条件には未確定事項が残る。

対象サービス

  • WeChat ミニプログラムによる乗車券情報表示、購入、WeChat Pay 決済
  • 購入後の顔登録・認証連携および乗車状態表示
  • 中間サーバと管理機能
  • 既存システムとの API 連携
  • 後続段階における乗車用 QR および座席指定の検討

想定发布方针

現時点では、顔認証を中心とする Ph.1 を先行し、乗車用 QR 対応を後続段階とする方向で整理する。サービス開始時期は 2027 年 1～3 月の複数口径が存在するため、2027 年 3 月前後を計画候補とし、要件定義完了後に正式確定する。座席指定は既存システムの大幅改修を伴う可能性があり、日程未定の後続範囲とする。

参照: SRC-001、SRC-002、SRC-003、SRC-008、SRC-009、METHOD-007、METHOD-005

2. プロジェクト体制

京成電鉄側

  • 業務部門、システム部門、プロジェクト責任者および既存システムベンダーで構成する想定。
  • 京成側責任者として宇野様が ACAP project binding に記録されているが、承認権限と分野別責任者は顧客確認を要する。
  • 既存システム改修、関連ベンダー調整、API 実装および UAT の主管を担う想定。

NSS側

  • プロジェクト統括、PM/PMO、設計、開発、インフラ、テスト、UI/UX および保守検討を担当する想定。
  • WeChat ミニプログラム側の開発コミュニケーション窓口を担う。
  • 社内の PM、技術ブリッジ、Tech Lead、UI/UX、QA の正式な担当範囲は要確認。

関連ベンダー

  • 日立製作所様: 顔認証および乗車券管理システムの担当候補。購入情報連携 I/F とシステム概要資料の提供が必要。
  • その他: 改札、座席管理、発券・印刷、ID/顧客管理等の担当会社と技術窓口は未確定。

3. スコープ

対象範囲

  • WeChat ミニプログラムの画面・画面遷移・簡体字対応
  • 乗車券購入、WeChat Pay 決済、注文・乗車状態表示
  • 購入後の顔登録 URL 連携、顔認証入場の支援
  • 中間サーバ、管理機能、開発・検証・本番環境
  • 乗車券管理、顔認証、改札等との対向 I/F 整理
  • 外部結合テストおよび京成電鉄 UAT 支援

対象外範囲

  • 既存システム側の改修実装は、原則として京成電鉄側主管の別作業とする想定。
  • インバウンドプロモーション施策そのものは本開発範囲外の候補。
  • 運用保守は別途範囲・見積・契約を確認する。
  • 座席指定は Ph.1 の必須範囲に含めず、後続段階として検討する。

Pending scope

  • 顔認証を完了しない利用者の有人窓口対応と例外条件
  • 返金・取消のシステム実装および状態連携
  • 管理機能、マーケティングデータ、行動ログの利用範囲
  • QR改札機対応の駅、設備、票種、リリース方式
  • 非機能、セキュリティ、運用保守、監視・SLA/SLO

フェーズ分割

  • Ph.1: 購入・決済、顔登録、顔認証入場、紙媒体による出場運用を中心とする候補。
  • Ph.1.1: 初期資料では顔写真撮影・アップロード追加段階として記載。Ph.1 への統合可否を確認する。
  • Ph.2/後続: 乗車用 QR の発行と QR改札機対応。座席指定は別判断とし、日程を固定しない。

参照: SRC-001、SRC-002、SRC-003、SRC-007、SRC-008、METHOD-005

5. 系统架构概要

  • WeChatミニプログラム: 利用者向け画面、購入、決済、顔登録接続、注文・乗車情報表示を提供する。
  • 中間サーバ: AWS 東京リージョン上の API Hub として、ミニプログラム、管理機能、既存システムを接続する想定。
  • 既存システム連携: 乗車券管理、顔認証、改札、発券・印刷、座席管理を対象候補とする。
  • API連携前提: 購入情報連携 I/F、顔登録 URL、認証結果、注文状態、例外処理、ログ・監視の仕様確定が必要。
  • データ管理: 個人情報、顔情報、WeChat ユーザー識別子、越境データの保存・削除・同意条件を要件定義で確認する。

AWS 東京リージョンの採用、API の具体的な本数・方向・責任分界は現時点では候補であり、ARC-001 と API-001 のレビューにより確定する。

参照: SRC-001、SRC-002、SRC-003、SRC-007、METHOD-005

6. 推進プロセス

  1. 要件定義: スコープ、UI/UX、業務フロー、対向 I/F、非機能要件を段階的に整理する。
  2. レビュー: 文書と課題票を用いて、業務整合性、仕様、既存システム影響を関係者で確認する。
  3. 合意形成: 未解決事項を残したまま approved とせず、合意範囲を版管理して更新する。
  4. 変更管理: 変更理由、影響 artifact、source、決定状態を change-log と registry に記録する。
  5. 課題管理: issue/risk/action/decision log を分離し、candidate、open、review_required を維持する。

参照: SRC-001、SRC-003、SRC-005、SRC-006、SRC-010、METHOD-007

9. コミュニケーション

  • 進捗定例: 週1回候補。進捗、課題、リスク、確認事項を共有する。
  • フォローアップ: 隔週候補。横断課題や意思決定待ちを扱う。
  • 個別ヒアリング: 京成電鉄各担当および関連ベンダーと随時実施する。
  • チャット / タスク管理: 京成電鉄指定ツールまたは Teams / Backlog 等を候補とし、正式運用を決定する。
  • ファイル共有: SharePoint を利用し、ファイル名に版番号または更新日を付与する。ルート管理と正式配置ルールは PMO が確認する。
  • 重要事項: 契約、承認、正式決定はメールまたは合意記録を残す。

参照: SRC-001、SRC-003、SRC-005、SRC-006、SRC-010

10. 確認事項

顧客待确认

  • プロジェクト承認者、予算・スコープ・UAT・リリース判定者

ベンダー待确认

  • 改札、印刷、座席管理を含む関連ベンダーと技術窓口
  • API 認証、頻度制限、ネットワーク、試験環境、ログ条件

NSS内部待确认

  • PM、Tech Lead、UI/UX、QA、インフラ、運用担当の正式アサイン
  • WBS、見積前提、非機能要件、テスト・移行・リリース計画
  • 顧客向け資料と社内検討資料の管理境界
  • method input の正式 artifact への採用可否
